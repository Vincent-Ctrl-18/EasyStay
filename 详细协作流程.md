# GitHub 三人协作详细流程（EasyStay / trip 通用）


## 1. 协作模型（强烈推荐）

采用 3 层分支：

- `main`：稳定可演示版本（禁止直接开发）
- `dev`：团队开发主线（所有功能先合并到这里）
- `feat/*`：个人功能分支（每人至少 1 个）

建议分支命名：

- A（后端）：`server`
- B（移动端）：`mobile`
- C（管理后台）：`admin`

---

## 2. 角色与职责

- **仓库管理员（建议你）**
  - 建立 `dev` 分支
  - 邀请协作者
  - 维护分支保护规则
  - 组织合并节奏
- **A 同学**：后端/API/数据库
- **B 同学**：移动端页面
- **C 同学**：管理后台页面


## 3. 每位成员首次接入（A/B/C 都执行）

> 每个人在自己的电脑执行，不要共用本地目录。

```bash
git clone https://github.com/Vincent-Ctrl-18/EasyStay.git  # 克隆远程仓库到本地
cd EasyStay                                                # 进入项目目录
git checkout dev                                           # 切换到团队开发主分支 dev
git pull origin dev                                        # 拉取远程 dev 最新代码
```

设置个人身份（非常重要，保证 GitHub 贡献记录正确归属）：

```bash
git config user.name "你的GitHub名字"
git config user.email "你的GitHub邮箱"
```

检查：

```bash
git config user.name
git config user.email
```

## 4. 创建个人分支并绑定远程

### A 同学

```bash
git checkout dev                      # 切回 dev，确保从最新主线开分支
git pull origin dev                   # 同步远程 dev
git checkout -b server         # 基于当前 dev 新建并切到个人分支
git push -u origin server      # 首次推送并建立上游跟踪关系
```

### B 同学

```bash
git checkout dev                      # 切回 dev
git pull origin dev                   # 同步远程 dev
git checkout -b mobile         # 新建并切换到 B 的功能分支
git push -u origin mobile      # 推送分支并绑定远程跟踪
```

### C 同学

```bash
git checkout dev                      # 切回 dev
git pull origin dev                   # 同步远程 dev
git checkout -b admin          # 新建并切换到 C 的功能分支
git push -u origin admin       # 推送分支并绑定远程跟踪
```

## 6. 固定流程

### 6.1 开工前

```bash
git checkout dev                      # 切到 dev，准备拉最新公共代码
git pull origin dev                   # 从远程更新 dev 到本地
git checkout 你的分支            # 切回你的功能分支
git merge dev                         # 把最新 dev 合并进你的分支
```

> 如果提示 `Already up to date.` 就表示同步完成。

### 6.2 开发 + 本地提交

```bash
git add .                                      # 把本次修改加入暂存区
git commit -m "feat(module): 完成xxx功能"      # 生成一次本地提交记录
git push                                       # 推送到你的远程分支（用于发 PR）
```

建议：每完成一个小功能就提交一次，不要憋大提交。

### 6.3 发起 PR（从个人分支到 dev）

GitHub 页面点击：

`Compare & pull request`

PR 模板建议：

- 做了什么（What）
- 为什么做（Why）
- 如何测试（How to test）
- 风险点（Risk）

### 6.4 代码评审后合并

- 至少 1 人 Review
- 通过后合并到 `dev`
- 合并后本地更新：

```bash
git checkout dev           # 切到 dev
git pull origin dev        # 拉取别人刚合并到 dev 的最新代码
```

## 7. 里程碑发布流程（dev -> main）

当你们准备“可演示版本”时：

1. 先确保 `dev` 最新且可运行
2. 由管理员发 PR：`dev -> main`
3. 全组验收通过后合并
4. 打标签（可选）

```bash
git checkout main          # 切到发布分支 main
git pull origin main       # 同步远程 main
git tag v1.0.0             # 打一个版本标签（里程碑快照）
git push origin v1.0.0     # 把标签推到远程
```

## 8. 提交规范（Conventional Commits）

统一使用：

- `feat:` 新功能
- `fix:` 修复问题
- `docs:` 文档更新
- `refactor:` 重构（无功能变化）
- `chore:` 工程杂项（配置、脚本、依赖）

示例：

```text
feat(mobile): 实现酒店列表无限滚动
fix(admin): 修复审核拒绝原因为空时提交报错
docs: 更新README启动步骤
```

## 9. 冲突处理（最实用）

场景：`git merge dev` 后冲突。

### 9.1 查看冲突文件

```bash
git status
```

### 9.2 打开冲突标记并手工处理

你会看到：

```text
<<<<<<< HEAD
你的代码
=======
dev 分支代码
>>>>>>> dev
```

按业务逻辑合并后，删除这些标记。

### 9.3 标记已解决并提交

```bash
git add .
git commit -m "fix: resolve merge conflicts with dev"
git push
```

## 10. 常见报错与对应解法

### 10.1 `src refspec main does not match any`

原因：当前分支没有任何提交。  
解法：

```bash
git add .
git commit -m "chore: init"
git push -u origin main
```

### 10.2 `remote origin already exists`

原因：远程已配置。  
解法（修改地址）：

```bash
git remote set-url origin https://github.com/Vincent-Ctrl-18/EasyStay.git
```

### 10.3 `rejected (fetch first)`

原因：远程有你本地没有的提交。  
解法：

```bash
git pull origin main --allow-unrelated-histories
# 解决冲突后再 push
```

### 10.4 `failed to push some refs`

常见原因：分支落后远程。  
解法：

```bash
git pull --rebase origin 当前分支
git push
```

## 11. 不建议做的事（踩坑清单）

- 不要直接在 `main` 开发
- 不要三个人同时长期改同一个文件（尤其 `App.tsx`、`api/index.ts`）
- 不要一次提交几千行且跨多个模块
- 不要把 `.env`、数据库大文件、`node_modules` 提交到仓库

---

## 12. 推荐 .gitignore（前后端项目）

在仓库根目录确保包含（如没有可补）：

```gitignore
node_modules/
dist/
.env
.env.*
*.log
.DS_Store
Thumbs.db
server/database.sqlite
server/uploads/
```

> `server/uploads/` 如果你们需要保留示例图，可改成只忽略大文件或保留占位文件。

## 14. 总结

### 管理员首次

```bash
git checkout main          # 切到主分支
git pull origin main       # 先确保 main 最新
git checkout -b dev        # 从 main 创建 dev 分支
git push -u origin dev     # 推送 dev 并建立跟踪
```

### 每位同学首次

```bash
git clone https://github.com/Vincent-Ctrl-18/EasyStay.git  # 克隆仓库
cd EasyStay                                                # 进入项目目录
git checkout dev                                           # 切到 dev
git pull origin dev                                        # 同步最新 dev
git checkout -b 你的分支                              # 新建个人分支并切换
git push -u origin 你的分支                           # 推送并绑定远程
```

### 每天开发

```bash
git checkout dev                      # 先拿最新公共代码
git pull origin dev                   # 更新本地 dev
git checkout 你的分支            # 回到个人分支继续开发
git merge dev                         # 把 dev 的更新合并到个人分支
# 开发完成后
git add .                             # 暂存改动
git commit -m "feat(scope): xxx"    # 提交改动
git push                              # 推送到远程个人分支
```
